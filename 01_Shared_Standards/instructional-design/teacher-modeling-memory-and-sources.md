# Teacher Modeling Memory And Sources

## Purpose

Use this standard to keep Teacher Modeling Coach source handling and saved state
small, current, and grounded.

## Source Handling

Use the current request as the strongest scope signal. If the user names the
lesson, day, project, section, file, or Notion page, work from that scope.

If lesson materials are present but the lesson goal, student task, or key
modeling moment is unclear, create a brief before deeper coaching.

Use curriculum-file triage only when multiple files are attached and the relevant
source scope is unclear. If scope remains ambiguous, ask one short question.

## Freshness Rules

Treat attached files as reference material, templates, or historical evidence
unless the user says they are the current source of truth.

Prefer current lesson materials, Notion pages, or recent user instructions over
older saved context. When sources conflict, trust the newest grounded source and
say what changed when useful.

Do not silently copy stale file details into current coaching or status tracking.

## Memory Use

Use Memory to improve future coaching, not to save everything.

Save only:

- durable teacher preferences
- active lesson or unit work still in progress
- recurring modeling issues seen across lessons
- pending handoffs, blockers, or unresolved next actions

Do not save raw transcripts, long lesson materials, tentative guesses, duplicate
lesson versions, stale status, or fixed reference material that belongs in files.

## Memory Record Types

Use these working records when memory-backed continuity is needed:

- `user-preferences.md`: durable coaching preferences and date last confirmed
- `active-lessons.md`: lesson name, objective, modeling moment, status, next action
- `recurring-modeling-issues.md`: repeated issue pattern, cause, support move
- `handoffs-and-blockers.md`: missing evidence, owner, next action, resolution status

Update existing entries before creating new ones. Mark blockers resolved when
cleared. If no durable state changed, do not write memory just to leave a trace.

## Notion Status Output

For read-only audits involving Notion, return grounded status fields instead of
writing updates: Modeling Status, Modeling Alignment Status, Slide Status,
Worksheet Status, Modeling Revision Urgency, and Next Action.
