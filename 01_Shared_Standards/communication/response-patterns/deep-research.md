# Deep Research Response Pattern

## Name

Deep Research

## Purpose

Support evidence-heavy questions while still producing a usable recommendation and clear next step.

## Status

experimental

## Default Modules

1. Research Question
2. Key Findings
3. Evidence
4. Recommendation
5. Next Step

## Max Length Guidance

- Target: 5 to 7 sections.
- Put the recommendation near the top or bottom, not hidden in the middle.
- Include citations or source references when research depends on external or repository evidence.
- Keep examples short unless the user asks for more.

## When To Use

- The user asks for deep research.
- The answer depends on source evidence.
- The user wants to compare how other projects solve a similar problem.
- The result should inform a later implementation decision.

## When Not To Use

- The user needs a quick classroom decision.
- The user is asking for a small rewrite or wording change.
- The work is an implementation report.
- The answer can be safely handled by Quick Decision.

## Example Response

**Research Question**
How do mature GitHub projects evolve reusable response or process templates?

**Key Findings**
- They keep templates small.
- They track maturity with statuses.
- They use issues or discussions for feedback.

**Evidence**
Use repository examples, docs, or citations here.

**Recommendation**
Start Agent OS response patterns as experimental files, not permanent governance.

**Next Step**
Test the first patterns for one week and record what works or gets in the way.

## Feedback Notes

```text
Did the evidence support the recommendation?
Was the research too long for the decision needed?
Was the next step actionable?
Keep / revise / reject:
```
